**✅ Google Cloud 請求代行事業者としての情報活用目的・整理**

以下に、これまでのテーブル情報（親アカウント / サブアカウント / リンクプロジェクト + ステータス変化）を、**請求代行事業者視点**で「どのような目的で活用するか」「誰が把握すべきか」「具体的なパターンとアクション」を体系的にまとめました。

### 1. 全体的な活用目的

この情報を定期的に取得・管理することで、以下の**3つの大きな価値**を生み出せます：

- **収益最大化・機会損失防止**（営業・CS視点） 
- **リスク管理・請求トラブル防止**（運用・バックオフィス視点）
- **顧客関係強化・サービス差別化**（全社視点）

### 3. 変化パターン別　目的・対応・営業機会

#### (1) **プロジェクト新規リンク**（最も重要な営業機会）

- **意味**: 顧客が新しくプロジェクトを作成or既存プロジェクトの請求先を自社サブアカウントに変更した
- **目的**: **追加売上の機会発見**
- **営業機会**:
  - 新規ワークロードの立ち上げ → インフラ設計支援、移行支援
  - 他社からの乗り換え → コスト削減提案
  - 利用規模拡大 → エンタープライズ契約や割引交渉
- **誰が把握すべき**: **営業担当（最優先）**、CS担当

#### (2) **プロジェクト アンリンク（UNLINKED）**

- **意味**: 請求先を解除した（他アカウントへ移動、または一時停止）
- **目的**: 解約リスク・チャーン検知
- **対応**:
  - 即時顧客ヒアリング（理由確認）
  - 他社への流出防止策
- **誰が把握すべき**: **CS担当、営業担当**

#### (3) **プロジェクト 再リンク（RE-LINK）**

- **意味**: 一旦解除したプロジェクトを再び自社請求先に繋いだ
- **目的**: 顧客の迷走・他社比較の把握
- **営業機会**: 価格競争力の再確認、サービス満足度調査、追加提案
- **誰が把握すべき**: **営業・CS**

#### (4) **billingEnabled = False**

- **意味**: プロジェクトはリンクされているが課金が無効化されている
- **目的**: 潜在的な請求漏れ防止、顧客のテスト/停止プロジェクトの把握
- **対応**: 長期化している場合は顧客に確認
- **誰が把握すべき**: CS、経理

#### (5) **サブアカウント closed**

- **意味**: 顧客のアカウント自体が閉鎖された
- **目的**: 解約確定、未収金リスク管理
- **誰が把握すべき**: 経理、CS、経営層

#### (6) **リンク数の急増（link_count増加）**

- **意味**: 顧客が積極的にGoogle Cloudを利用拡大中
- **目的**: **ホットリード**として営業強化
- **誰が把握すべき**: 営業（優先度高）

#### (7) **前月請求金額が0円**（prev_month_cost = 0）

- **意味**: 課金は有効だが前月の実際の利用費用がゼロ（テスト中・休眠・誤リンクの可能性）
- **目的**: 休眠プロジェクトの把握、顧客の利用状況確認
- **注意点**: 0円でも必ずしも非稼働ではない（コミット済みのサポートプラン費用など請求代行に含まない費用もある）。あくまで参考情報として活用する。
- **対応**: 長期間0円が続く場合は顧客にヒアリング（誤リンクの整理、利用促進）
- **誰が把握すべき**: CS担当、営業担当

#### (8) **累積未課金プロジェクト**（ever_billed = FALSE）

- **意味**: リンク開始から現在まで一度も課金実績がないプロジェクト
- **目的**: 未活用リソースの発見、誤リンクの検出
- **対応**: 顧客に確認の上、不要なリンクを整理 or 活用促進の提案
- **誰が把握すべき**: CS担当

#### (9) **未課金プロジェクトの課金開始**（billing_newly_started = TRUE）

- **意味**: これまで一度も課金がなかったプロジェクトが初めて課金された（本格稼働開始）
- **目的**: **最も重要な営業シグナルの一つ**。顧客がGoogle Cloudの本格利用を開始した瞬間を捉える。
- **営業機会**:
  - 利用開始直後の手厚いオンボーディング支援
  - コスト最適化・構成レビューの提案
  - 将来的な利用拡大に向けた関係構築
- **通知**: エラー通知とは別の専用Slackチャンネルにリアルタイム通知することを推奨
- **誰が把握すべき**: **営業担当（最優先）**、CS担当

### 4. 役割別　把握すべき情報と頻度

| 役割 | 把握すべき主な情報 | 確認頻度 | 目的・期待アクション |
|----------------|----------------------------------------|--------------|----------------------|
| **営業担当** | 新規リンク、再リンク、リンク数増加、**課金開始（billing_newly_started）** | 毎日 | 営業機会の即時フォロー |
| **CS担当** | アンリンク、billingEnabled=false、closed、**前月0円プロジェクト、累積未課金プロジェクト** | 毎日 | 顧客満足度維持、リスク早期発見 |
| **経理・管理部** | 全サブアカウントのopen状態、請求可能額 | 月1回 | 請求処理、未収リスク管理 |
| **技術担当** | プロジェクト数・利用傾向 | 週1回 | 技術支援ニーズの把握 |
| **経営層** | 全体リンク数推移、チャーン率 | 月1回 | 事業戦略、KPI管理 |

### 5. 運用として推奨する仕組み

- **毎日実行**: 同期スクリプトをCloud Scheduler + Cloud Functionsで自動化
- **変化検知アラート**:
  - 新規リンク → Slack/メールで営業に通知
  - アンリンク → CSに優先通知
- **ダッシュボード**: Looker StudioやBigQuery + Data Studioで可視化
- **月次レビュー**: 営業・CS合同で「動きのあった顧客」レビュー会議

### まとめ：最も価値が高い使い方

この情報を\*\*「静的な請求データ」ではなく「顧客行動のリアルタイムセンサー」\*\*として扱うことです。

特に\*\*「プロジェクトの新規リンク」**は、顧客が「Google Cloudで何か新しいことを始めようとしている」明確なシグナルです。\
請求代行事業者だからこそ早期に捉えられる**最大の営業機会\*\*と言えます。

このテーブルと同期仕組みをしっかり運用することで、\
「ただの請求代行」から「顧客のGoogle Cloud利用拡大のパートナー」へのポジションアップが実現できます。

必要であれば、**役割別ダッシュボードのイメージ**や**アラート設計**、**KPI例**も追加で整理します。
